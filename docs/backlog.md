## 1. Пароль SMB виден через `ps aux` на macOS

**Где:** `app/smb_mount.py`, `_mount_macos()` (строка ~182).

**Проблема:** `mount_smbfs` не поддерживает файл с учётными данными (в отличие от
`mount.cifs` на Linux, который читает пароль из `credentials=<tmpfile>`).
Флаг `-N` (чтение пароля из stdin) ненадёжен в разных версиях macOS, поэтому
пароль передаётся прямо в URL: `smb://user:pass@host/share`. Любой другой
локальный непривилегированный пользователь на той же машине может увидеть
пароль сервисного аккаунта SMB через `ps aux` в момент (пере)монтирования
при старте/reload сервера.

**Почему отложено:** это ограничение самого macOS/`mount_smbfs`, а не баг в
коде приложения — эквивалента `credentials=file` для `mount_smbfs` не
существует. Обходные пути (запуск `mount_smbfs` через обёртку, скрывающую
аргументы от `ps`, или монтирование через `expect`/AppleScript-Keychain)
существенно усложняют macOS-ветку кода ради узкой угрозы (нужен другой
локальный пользователь на том же macOS-хосте — не сетевой вектор).

**Что сделать при возврате:** оценить, стоит ли переключиться на
Keychain-based монтирование (`osascript` + Keychain item) или документировать
как принятый риск для однопользовательских macOS-хостов; на Linux эта
проблема уже не актуальна благодаря `credentials=`.

## 2. DST-неоднозначность в дисковом кэше индекса файлов

**Где:** `app/cache.py`, строки ~33 и ~63 (`af.start_dt.timestamp()` /
`datetime.fromtimestamp(f["start"])`).

**Проблема:** naive-datetime (без таймзоны) конвертируется в Unix-timestamp
и обратно через локальную таймзону. В час перевода с летнего на зимнее время
(fall-back) один и тот же naive-datetime неоднозначен — конвертация туда и
обратно может дать сдвиг на час.

**Почему отложено:** проявляется максимум дважды в год, только для файлов,
чьё время записи попадает точно в этот "двойной" час, и только влияет на
кэш (при следующем `refresh()`/rescan данные пересчитываются заново из
реальных файлов на SMB/диске — кэш не единственный источник истины).
Правильное исправление требует протаскивания таймзоны через `AudioFile`,
`ChannelIndex` и весь путь сохранения/загрузки кэша — заметный рефакторинг
ради очень редкого и самоисправляющегося случая.

**Что сделать при возврате:** либо хранить `start_dt`/`end_dt` как
timezone-aware (UTC) по всей цепочке `file_index.py` → `cache.py` →
`models.AudioFile`, либо явно задокументировать и принять двухдневный сдвиг
как известное ограничение.

## 3. Битрейт MP3/AAC определяется только по первым 64 КБ файла

**Где:** `app/audio_probe.py`, строки ~96 и ~123–130 (`mutagen.mp3.MP3` /
`mutagen.aac.AAC` парсят только `raw` — первые 64 КБ, обходной путь вокруг
дорогого seek по SMB).

**Проблема:** для VBR-кодированных файлов битрейт из первых 64 КБ не
обязательно совпадает со средним битрейтом всего файла — значение,
сохранённое в `detected_params.yaml` и используемое для оценки длительности
(`_estimate_duration` в `file_index.py`) и как значение по умолчанию при
экспорте, может быть систематически неточным.

**Почему отложено:** это сознательный компромисс (см. комментарий в самом
`audio_probe.py`) ради избежания полного скачивания/сика по SMB на каждый
пробник; станции в этом проекте почти всегда вещают в CBR, где ошибка
минимальна. Полное решение (чтение всего файла или сэмплирование в
нескольких местах файла) увеличивает нагрузку на SMB на каждый детект.

**Что сделать при возврате:** если появятся станции с заметным VBR-контентом,
рассмотреть сэмплирование битрейта в 2–3 точках файла (начало/середина/конец)
вместо одной, усредняя результат, либо явно предупреждать в UI, что битрейт
приблизительный для таких каналов.
