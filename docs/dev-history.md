## 0. Создание приложения с нуля

*Первая сессия, Mac. Проект тогда назывался «radio-monitor», v0.1.001.*

> Мне нужно веб-приложение для поиска нужного времени в записях контроля радиоэфира, выделения и экспорта фрагмента. Радиостанций и каналов может быть несколько, настройки в YAML. На экране: области прокрутки (секунды, минуты, часы, дни, месяцы), выбор канала с поиском, плейлог, список сегментов.

**Создана вся базовая архитектура:**

| Модуль | Функция |
|--------|---------|
| `smb_client.py` | Единый API для локальных путей и SMB (`smbprotocol`) |
| `file_index.py` | In-memory индекс файлов, polling, WS-уведомления |
| `audio.py` | Стриминг и экспорт через ffmpeg (конкатенация, обрезка) |
| `playlist.py` | Чтение CSV с гибким маппингом полей |
| `main.py` | REST API + WebSocket |

**`static/timeline.js`** — canvas-таймлайн, 5 рядов (секунды → месяцы), синхронизированы через `centerTime`. Синяя подсветка = есть аудио, жёлтая = выделенный сегмент.

**Первый коммит** — инициализирован git-репозиторий, перенесён на GitHub, продолжение разработки на Windows.

```python
async def _try_scan(self, idx, max_retries=2, timeout=30.0) -> bool:
    for attempt in range(1, max_retries + 1):

            return True
        except asyncio.TimeoutError: ...
```

### SMB-монтирование на Linux, диагностика

> Сетевая папка не монтируется на Ubuntu 26.04.

Причина: `smb_mount.py` не передавал `sec=` в `mount.cifs`. Ubuntu 26.04 (ядро 6.11+) требует `sec=ntlmssp`.

```python
sec = "krb5" if smb.auth_protocol == "kerberos" else "ntlmssp"
base_options = [..., f"sec={sec}"]
```

`tools/smb_diag.py` — реальный `mount.cifs`, проверка Kerberos-тикетов.

> STUDIO 21 — очень долгая буферизация. Диагностика.

```python
dl_tasks = [asyncio.create_task(_stage_one(i, af)) for i, af in enumerate(files)]
for i, (af, dl_task) in enumerate(zip(files, dl_tasks)):
    local_path = await dl_task   # ждём только этот файл
    # ffmpeg стартует немедленно
```

## 7. Code review и исправление ошибок

*Сессия 2026-06-12, v1.2.042. Коммит `2295230`.*

### Методология

Проведён многоугольный code review изменений рабочего дерева: параллельный запуск агентов по трём углам (diff scan, removed-behavior audit, cross-file tracer) с последующей независимой верификацией каждого кандидата (CONFIRMED / PLAUSIBLE / REFUTED).

### Критические исправления

**1. Несовместимость naive/aware datetime — `app/cache.py`**

Предыдущее изменение добавило `tz=timezone.utc` при загрузке дат из дискового кэша (`datetime.fromtimestamp(ts, tz=timezone.utc)`). Это породило несовместимость: `_scan_date()` в `file_index.py` создаёт naive-datetime из имён файлов, route handlers в `main.py` передают `datetime.fromtimestamp(start)` без tz. При совместном нахождении aware и naive объектов в одном `SortedList` — `TypeError` при любом аудио-запросе после загрузки кэша.

Исправление: откат `cache.py` к naive-datetime. Вся система единообразно использует локальное время.

**2. Неверный escape одиночных кавычек в ffconcat — `app/audio.py`**

В `stream_audio` был применён SQL-стиль удвоения кавычек (`'` → `''`), который не является валидным escape для ffconcat: демультиплексор воспринимает первую `'` как конец имени файла. Правильный escape — `'` → `\'`.

Исправление: `.replace("'", "\\'")` в обоих местах.

### Прочие исправления

**3. Избыточный `os.chmod` после `mkstemp` — `app/smb_mount.py`**

`tempfile.mkstemp` всегда создаёт файл с правами 0600 вне зависимости от `umask` процесса. Явный `os.chmod(creds_path, 0o600)` был лишним и удалён.

**4. Двойной `import os` внутри функции — `app/auth.py`**

`os` не был импортирован на уровне модуля, поэтому появился дважды внутри `_save_sessions()` — один раз в `try`-блоке, второй в `except`. Вынесен в заголовок модуля.

## 9. Замена argon2 на PBKDF2-SHA256 (stdlib)

*Сессия 2026-06-12, v1.2.044.*

### Проблема

На Python 3.14 `cffi 2.0.0` не может загрузить C-расширение `argon2-cffi-bindings`: при `from argon2 import PasswordHasher` возникает не `ImportError`, а ошибка инициализации `.so`. Результат — вход с Argon2-хэшем невозможен на продакшн-сервере.

### Решение

Отказ от внешней зависимости `argon2-cffi` в пользу `hashlib.pbkdf2_hmac` (Python stdlib, работает на LibreSSL и OpenSSL, CPython любой версии).

**Формат хэша** — `$pbkdf2$sha256$<iterations>$<salt_base64>$<hash_base64>`  
**Параметры** — PBKDF2-SHA256, 600 000 итераций (рекомендация OWASP 2023 для PBKDF2-SHA256), соль 32 байта, dklen 32 байта.

### Изменения

| Файл | Что изменилось |
|---|---|
| `app/auth.py` | `_hash_pbkdf2()`, `_verify_pbkdf2()`; новое поле `password_pbkdf2` с приоритетом 1 |
| `app/auth.py` | Argon2-блок сохранён как legacy (приоритет 2), с корректным fallback при ошибке импорта |
| `tools/hash_password.py` | Переписан на PBKDF2, больше не требует argon2-cffi |
| `requirements.txt` | Строка `argon2-cffi>=23.1.0` удалена |
| `config/users.yaml` | Поля `password_argon2` заменены на `password_pbkdf2` |

### Приоритет полей в users.yaml

1. `password_pbkdf2` — PBKDF2-SHA256 (stdlib, рекомендуется)
2. `password_argon2` — Argon2id (legacy, argon2-cffi)
3. `password` — открытый текст (только обратная совместимость)

### SMB pipe-стриминг: мгновенный старт воспроизведения (v1.2.057)

**Проблема:** при воспроизведении с SMB-шары браузер получал первый аудиочанк только после того, как весь файл скачивался целиком во временную папку (для часового эфира — 200–300 МБ, задержка десятки секунд).

**`app/audio.py`:**

- Для чистых SMB-каналов (без локального монтирования) `stream_audio` теперь использует **pipe-режим**: файл открывается через `smb.open_file()`, хэндл сразу seekируется до нужного байтового оффсета, данные читаются чанками (64 КБ) и передаются в **stdin** ffmpeg через `asyncio` pipe

### Исправление воспроизведения старых файлов при смене ОС сервера (v1.2.065)

**Причина:** при переносе сервера с Windows на Linux диск-кэш `file_index` содержал записи с `af.path = \\server\share\...` (Windows UNC). На Linux такие пути не существуют. `_stage_one` возвращал путь напрямую → ffmpeg: `No such file or directory`. Свежие файлы работали, т.к. были переиндексированы с корректным Linux-путём.

**Исправления в `app/audio.py`:**

1. **`af.rel_path` вместо `_rel_from_af`** — реальный путь из индекса вместо реконструированного из шаблонов конфига. Расхождение возможно при изменении конфига после сканирования.

2. **`_is_local_path_usable(path)`** — на POSIX возвращает `False` для `\\...` (Windows UNC). В `_stage_one`: если `af.path` недоступен на текущей ОС — fallback на SMB-загрузку.

3. **`_drain_stderr_local`** — параллельный дренаж stderr ffmpeg в staging-пути. Предотвращает потенциальный deadlock (~64 KB pipe). Ошибки из stderr → INFO, прогресс → DEBUG.

### Устранение задержки 5-6 с перед стартом аудио в SMB pipe-режиме (v1.2.066)

**Причина:** при воспроизведении через SMB pipe ffmpeg получает вход как `pipe:0`. Для pipe-входов ffmpeg использует значения по умолчанию: `analyzeduration = 5 000 000 мкс` (5 секунд) и `probesize = 5 000 000 байт`. Это заставляет ffmpeg буферизовать 5 секунд или 5 МБ данных перед тем, как начать вывод — независимо от позиции в файле. Поэтому задержка была одинакова и в начале, и в середине, и в конце 60-минутного файла.

**Исправление в `app/audio.py`:**

- В `_pipe_smb_segment`: добавлены флаги `-analyzeduration 0 -probesize 32` перед `-i pipe:0`. Поскольку формат указан явно через `-f mp3` / `-f adts` / `-f wav`, анализ не нужен — ffmpeg начинает вывод почти мгновенно.

- В `_open_file`: замена `fh.seek(0, 2) + fh.tell()` на `smb.getsize()` для получения размера файла. `seek(0, SEEK_END)` на некоторых реализациях SMB может читать весь файл последовательно. `smb.getsize()` использует `smbclient.stat().st_size` — один быстрый SMB QueryInfo-запрос.

### Прямой SMB-стриминг без ffmpeg для MP3/AAC (v1.2.067)

**Причина:** при copy mode (MP3→MP3, AAC→AAC) ffmpeg не делает ничего полезного — только копирует байты. Но запуск процесса ffmpeg, настройка stdin/stdout pipes и даже минимальный `analyzeduration` всё равно добавляют задержку. Оставшиеся 4 секунды перед стартом объяснялись накладными расходами на сам SMB-вызов `open_file` + ffmpeg-запуск + ожидание первых данных в pipe.

**Исправление в `app/audio.py`:**

- Добавлен `_direct_smb_segment`: открывает SMB-файл, seekит к байтовому смещению, читает чанками по 512 KB (было 64 KB → меньше round-trips) и льёт байты напрямую в HTTP-ответ. Никакого ffmpeg.
- В `stream_audio`: если `_use_pipe` и `copy_mode` и формат MP3 или AAC — используется `_direct_smb_segment`. WAV исключён (браузер требует RIFF-заголовок с нулевого смещения).
- MP3 и AAC (ADTS) — самофреймирующиеся форматы: браузер воспроизводит их с любого байта без заголовка-контейнера.

### Fix: сброс кеша secret.yaml при /api/reload (v1.2.071)

**Причина:** `_secrets` в `app/config.py` — модульный кеш, заполняемый один раз при первом обращении и никогда не сбрасываемый. При `POST /api/reload` станции и плейлоги парсились заново (`load_stations()` / `load_playlists()`), но `_parse_smb()` резолвил `secret: N` по устаревшему, закешированному с момента старта содержимому `secret.yaml`. ## 12. AAC: воспроизведение, автоопределение формата и битрейта; плейлоги: маска файлов и BOM

*Сессия 2026-07-07, ветка `fix/aac`.*

### Fix: ошибка воспроизведения AAC — неверное имя ffmpeg-демаксера

**Причина:** в `_pipe_smb_segment` (`app/audio.py`) для входного потока `.aac`-файлов передавался флаг `-f adts`. В ffmpeg `adts` — это имя **мьюксера** (выходного формата); демаксер для сырого ADTS AAC-потока называется `aac`. В результате ffmpeg падал с `Unknown input format: 'adts'` и не отдавал ни байта аудио. Воспроизведён и подтверждён реальным запуском ffmpeg с обоими значениями `-f`.

**Исправление:** `in_fmt` для расширения `aac` заменён на `"aac"`.

### Автоопределение контейнера AAC (ADTS vs MPEG-TS) при сканировании (v1.2.072)

**Причина:** часть источников (HLS-TS захват) сохраняет `.aac`-файлы не как сырой ADTS-поток, а как всё ещё MPEG-TS-контейнер под расширением `.aac`. Демаксер для них должен быть `mpegts`, а не `aac` — иначе даже с исправленным именем демаксера файл не воспроизводится.

**Реализация:**

- `app/audio_probe.py` — `_detect_aac_container()`: сигнатурный анализ первых байт файла (пропуская опциональный заголовок ID3v2): sync-слово ADTS `0xFFF` → `"aac"`, повторяющийся каждые 188 байт sync-байт `0x47` → `"mpegts"`. `probe()` возвращает найденный формат в поле `in_format`; битрейт через `mutagen.aac.AAC` определяется только для реального ADTS (для MPEG-TS попытка обречена и пропускается).
- `app/models.py` — новое поле `ChannelConfig.aac_input_format`.
- `app/file_index.py` — `_apply_detected()` применяет `in_format` к каналу; `probe_channel()` пробует канал хотя бы один раз даже при уже заданном битрейте, если формат AAC-контейнера ещё не определён.
- `app/audio.py` — `_pipe_smb_segment` использует `channel.aac_input_format or "aac"` вместо жёстко заданного значения. Прямое побайтовое копирование в браузер (`_use_direct`, без ffmpeg) отключено для каналов с определённым `mpegts`-контейнером — сырые TS-байты не воспроизводимы браузером как ADTS, такие каналы всегда идут через ffmpeg (демакс + ремакс в ADTS).
- Определённый формат сохраняется в `config/detected_params.yaml` (тот же файл, что и битрейт) и переживает перезапуск — повторное определение не требуется, пока не будет запрошен принудительный full rescan.

Проверено: сгенерированы тестовые файлы через `ffmpeg -f adts` и `ffmpeg -f mpegts`, `_detect_aac_container` и `probe()` корректно различают оба случая (ADTS → `bitrate`+`in_format=aac`, TS → только `in_format=mpegts`).

### Fix: неверная длительность AAC-файлов на таймлайне (v1.2.072)

**Причина:** для `.aac`-каналов автоопределение битрейта никогда не запускалось (`audio_probe.probe()` поддерживал только `mp3`/`wav`) — длительность файла всегда оценивалась по статичному/дефолтному битрейту (`_estimate_duration` в `app/file_index.py`), из-за чего 10-минутный файл занимал на таймлайне 7:50.

**Исправление:** `audio_probe.probe()` и `file_index._apply_detected()` / `probe_channel()` расширены на `aac` по той же схеме, что уже была для `mp3` (см. выше) — реальный битрейт из `mutagen.aac.AAC` теперь применяется к каналу и используется при оценке длительности.

### Новая настройка: полное пересканирование аудио при старте (v1.2.072)

`config/settings.yaml` → `indexing.rescan_all_on_startup` (по умолчанию `false`). При `true` `file_index.initial_scan()` пропускает быструю загрузку из дискового кеша и на каждом запуске/`/api/reload` принудительно передетектит битрейт/формат и пересканирует все каналы (`app/file_index.py`, `app/main.py`) — аналогично ручной кнопке «Пересканировать» для канала, но для всех каналов сразу.

### Fix: модуль плейлогов не видел источники с расширением, отличным от `.log`

**Причина 1:** `_check_source_health()` в `app/playlist.py` жёстко фильтровал файлы по `.log`, игнорируя `source.file_mask`. Плейлог с `file_mask: "%Y-%m-%d.csv"` (пример — `loveradio.yaml`) всегда получал статус «no .log files in folder», даже когда `.csv`-файлы были на месте.

**Причина 2:** тестовый CSV содержал UTF-8 BOM. `_parse()` декодировал байты, но не снимал BOM — он приклеивался к первой ячейке заголовка (`﻿EventTime`), из-за чего сопоставление имён колонок ломалось и файл давал 0 распознанных записей молча.

**Исправление:**

- `_check_source_health()` — расширение вычисляется из `Path(source.file_mask).suffix`, а не захардкожено.
- `_parse()` — `text.lstrip("﻿")` после декодирования.

Проверено на реальном примере (`loveradio.yaml` + образец CSV на 29 записей): `_check_source_health` → `{"ok": True}`, `_parse` → все 29 записей распознаны корректно.

## 14. Временный сбой SMB/Kerberos стирал весь кэш файлов канала

*Сессия 2026-07-08, ветка `fix/cache`.*

### Fix: после рестарта сервера канал показывал 0 файлов, пока не нажмёшь «Пересканировать»

**Симптом:** сразу после рестарта дисковый кэш загружался мгновенно («Loaded 312 files from disk cache»), но канал `radio_energy-hls` (SMB c `auth_protocol: kerberos`) тут же показывал 0 файлов — реальные записи появлялись только после ручного «Пересканировать аудио текущего канала». В логе — `SpnegoError`/«No Kerberos credentials available» на сканировании **каждой** из 90+ дат подряд.

**Причина:** `smb_client.scandir()` намеренно пробрасывает исключение при ошибке авторизации/подключения (в отличие от «папка ещё не создана», которая тихо возвращает `[]`) — так и задумано, чтобы вызывающий код пометил канал недоступным. Но `ChannelIndex.refresh()` (`app/file_index.py`) ловил это исключение **на уровне каждой отдельной даты** внутри цикла и просто логировал на DEBUG, продолжая сканирование:
```python
for d in dates:
    try:
        current.update(self._scan_date(d))
    except Exception as exc:
        log.debug(...)   # проглочено молча
```
Ошибка Kerberos-тикета — на уровне хоста, а не даты, поэтому повторялась на всех 91 датах разом, `current` оставался полностью пустым словарём, а diff-логика дальше трактовала «сканирование вернуло 0 файлов» как «все ранее известные файлы удалены» — стирая из индекса всё, что было загружено из дискового кэша при старте. Раз исключение не выходило из `refresh()`, `_try_scan()`/`_poll_loop()` тоже не видели ошибку: канал не помечался «unreachable», лог `Channel %s unreachable` не печатался — сбой был полностью не виден на уровне INFO.

**Исправление:** `ChannelIndex.refresh()` теперь отличает «папка не создана» (безопасно считать пустой) от настоящей ошибки подключения/авторизации. Если хоть одна дата упала с настоящей ошибкой, diff **не применяется** — существующий индекс (из кэша или предыдущего успешного скана) остаётся нетронутым, а последняя ошибка пробрасывается наверх, чтобы `_try_scan()`/`_poll_loop()` штатно пометили канал недоступным (retry, `ERROR: Channel unreachable`, авто-восстановление на следующем цикле опроса) — вместо молчаливого обнуления файлов.

Проверено юнит-тестами: (1) все даты падают с ошибкой авторизации → предзагруженные 3 файла в индексе остаются нетронутыми, ошибка корректно пробрасывается; (2) все даты успешно сканируются и находят 0 файлов (реальное удаление) → устаревшая запись корректно удаляется — старое поведение для легитимного случая не сломано.

### Доп.: пауза между повторами скана, чтобы уменьшить саму гонку с Kerberos-тикетом

Фикс выше делает сбой видимым и не разрушительным, но не устраняет саму гонку: тикет обычно появляется в `/tmp/krb5cc_<uid>` асинхронно (внешний `kinit`/cron), и на обычном рестарте первый скан всё ещё стартует раньше, чем тикет готов — красный индикатор при рестарте по-прежнему возможен, лечится только следующим циклом опроса или ручным «Пересканировать».

`FileIndexManager._try_scan()` делал 2 попытки **без паузы между ними** — при детерминированной (не по таймауту) ошибке вроде `SpnegoError` вторая попытка проваливалась ровно так же, как первая, спустя миллисекунды, не давая тикету времени появиться. Добавлена пауза `retry_delay=3.0` с между попытками (не после последней) и увеличено число попыток `max_retries` 2 → 3 — то есть до 2 пауз по 3 с на канал, прежде чем канал будет помечен недоступным. Если тикет появляется в течение этого окна, скан молча сходится в норму без участия пользователя.

Проверено юнит-тестами на реальном таймере: (1) сбой на всех 3 попытках → ровно 2 паузы по `retry_delay`, без задержки после финальной проваленной попытки; (2) сбой на 1-й попытке, успех на 2-й → возврат сразу после успеха, без лишней паузы.

## 15. Перемотка во время воспроизведения молчала после выделения фрагмента

*Сессия 2026-07-08, ветка `fix/aac`.*

### Fix: перемотка на позицию после старого выделения отправляла start > end — тишина без ошибки

**Симптом:** во время воспроизведения AAC (`radio_energy-hls`) перемотка по таймлайну сдвигала плейхед, но звук пропадал; помогал только стоп-старт. В логе — запросы вида `Stream radio_energy-hls  2026-07-08 06:07:52 → 05:57:47` (время начала **позже** времени конца) и `06:29:25 → 05:57:47`.

**Причина:** `_seekPlayback()` (`static/app.js`) при перемотке во время воспроизведения брал конец диапазона из текущего выделения (`Timeline.getSelection().end`), если оно есть, без проверки, что цель перемотки всё ещё раньше этого конца:
```js
const endTs = (selE !== null) ? selE : ts + 3600;
```
Если пользователь ранее выделял фрагмент (In/Out) с фиксированным концом, а затем перематывал **дальше** этого конца, `endTs` оставался равен старому `selE` — запрос уходил с `start > end`. `file_index.files_for_range()` на такой диапазон не находит файлов и возвращает пустой ответ без ошибки: плеер получает пустой поток, таймлайн уже сдвинут анимацией — звука нет, ничего не залогировано как сбой. Баг не специфичен для AAC — тот же путь кода для любого формата, просто чаще всего замечался на AAC-тестовом канале с активным выделением.

`startPlay()` уже содержал верную защиту (`ts <= selE`), которой не было в `_seekPlayback()` — стоп-старт срабатывал именно потому, что `startPlay()` пересчитывает `endTs` заново с этой проверкой.

**Исправление:** та же защита добавлена в `_seekPlayback()`:
```js
const endTs = (selE !== null && ts <= selE) ? selE : ts + 3600;
```

Проверено в браузере: с закэшированным (устаревшим) `app.js` воспроизведён баг один-в-один — перемотка на `06:07:52` при выделении с концом `05:57:47` (тот же таймстамп из лога) даёт `start > end`. После применения фикса тот же вызов корректно откатывается на `ts + 3600` (`start < end`), перемотка в пределах ещё не пройденного выделения по-прежнему уважает его конец.

## 16. Фрагменты с перерывами воспроизводились/отображались не в то время

*Сессия 2026-07-14, ветка `fix/silence`.*

### Fix: пропуски между фрагментами не озвучивались тишиной — плейхед расходился с реальным содержимым

**Симптом:** на одном из wav/PCM-каналов (запись фрагментами — только при наличии сигнала) при воспроизведении выделения, пересекающего перерыв между фрагментами, положение плейхеда на таймлайне переставало соответствовать тому, что слышно.

**Диагностика:** реальные файлы канала за проблемный день проверены через `ffprobe` — имена файлов (время старта) и оценка длительности в `file_index.py` (`_estimate_duration`, размер WAV минус 44-байтный заголовок) оказались точны, `end_dt` каждого фрагмента почти впритык совпадает с `start_dt` следующего или с реальным перерывом записи. Индексация не виновата.

Причина — в `app/audio.py`: `stream_audio()` и `export_audio()` конкатенировали байты найденных фрагментов подряд, **не добавляя тишину на месте перерыва** между `af.end_dt` предыдущего файла и `af.start_dt` следующего. Итоговый поток оказывался короче запрошенного диапазона на суммарную длину пропусков. При этом плейхед в `static/app.js` (`_startPlayheadAnimation`) двигается по формуле `playFromTs + elapsed` — линейно от реального времени с начала воспроизведения, в предположении непрерывности звука. Как только поток «перепрыгивал» пропуск, реально звучащий фрагмент убегал вперёд/назад относительно того, что показывал таймлайн, и расхождение росло с каждым следующим пропуском.

**Исправление** (`app/audio.py`):

- Добавлены `_silence_cmd()`/`_silence_segment()` — генерация тишины нужной длительности через `ffmpeg -f lavfi -i anullsrc=...`, закодированной в том же формате/битрейте, что и соседние сегменты (для WAV — `pcm_s16le`, для MP3/AAC copy-режима — родной кодек/битрейт канала).
- В `stream_audio()` (оба режима — SMB pipe и локальный/примонтированный) перед каждым сегментом вычисляется разрыв между концом предыдущего фрагмента и началом следующего (`cursor_dt` vs. `af.start_dt`); при разрыве > 50 мс между сегментами вставляется тишина соответствующей длины. Аналогично — тишина перед первым фрагментом, если выделение начинается раньше него.
- В `export_audio()` та же логика реализована через физические временные WAV/MP3/AAC-файлы тишины (`_write_silence_file()`), добавляемые в `ffconcat`-список между реальными фрагментами — родной формат файла тишины совпадает с форматом канала, поэтому список остаётся однородным и для `-c copy`, и для перекодирования.
- Попутно убрана функция `_stage_files()` (обёртка над `_stage_one()`, теряла привязку файла к его `AudioFile` при параллельной загрузке — из-за этого нельзя было посчитать разрывы); `export_audio()` теперь сам собирает пары `(AudioFile, локальный_путь)`.

Проверено: симуляция реального выделения (10:31:30–10:36:00, три фрагмента + два внутренних пропуска, файлы взяты напрямую с проблемного канала) через `ffmpeg`/`ffprobe` — итоговая длительность конкатенации с тишиной 270.04 с при ожидаемых 270.0 с (расхождение — округление ffmpeg до кадра), без вставки тишины было бы ровно на суммарную длину пропусков короче.

### Регрессия: воспроизведение зависало на тишине / переставало продолжаться при перемотке через пропуски

**Симптом (обнаружен после деплоя фикса выше):** при перемотке таймлайна через участки с пропусками воспроизведение переставало продолжаться нормально — вместо ожидаемого содержимого играла долгая тишина, либо звук вовсе пропадал, и казалось, что плеер «не двигается дальше» / застревает.

**Причина:** `stream_audio()` изначально добавлял тишину также **после последнего найденного фрагмента, вплоть до `end`** («trailing»-пропуск). Но при обычном воспроизведении без явного выделения фронтенд (`startPlay()`/`_seekPlayback()` в `static/app.js`) всегда запрашивает `end = ts + 3600` — мягкий условный «запас на час вперёд», а не реальную границу. Если реально записанный материал заканчивался раньше (обычная ситуация у живого эфира — вперёд записано меньше часа или впереди ещё не записанный момент), исправление честно генерировало тишину **вплоть до часа** вместо того, чтобы поток естественно заканчивался, как раньше. Пока эта фоновая тишина «играла», перемотка либо попадала в неё же, либо, выйдя за пределы часового окна совсем без файлов, получала пустой ответ — субъективно это выглядело как зависание/невозможность продолжить воспроизведение.

Внутренние пропуски (между двумя фрагментами, уже найденными в границах запроса) и пропуск перед первым фрагментом такой проблемы не создают — они ограничены реальными записанными данными хотя бы с одной стороны. «Хвостовой» пропуск ограничен только произвольной константой `+3600`, поэтому именно он мог порождать почти часовую тишину.

**Исправление** (`app/audio.py`): генерация тишины после последнего фрагмента убрана из обоих режимов `stream_audio()` — поток снова естественно заканчивается там, где заканчиваются реальные данные (браузер получает `ended`, плеер штатно останавливается). В `export_audio()` эта же логика **оставлена** — там `start`/`end` всегда берутся из явного выбора пользователя (Mark In/Out или сохранённый фрагмент, не более `MAX_SEGMENT_SEC` = 3 ч), а не из условного часового запаса, поэтому досозданная тишина там семантически корректна и безопасно ограничена.

### Независимый баг (не связан с тишиной выше): перемотка внутрь WAV-фрагмента давала полную тишину

**Симптом:** пользователь прислал серверный лог (`2026-07-14.log`) с реальным продакшн-запуском (Linux, ffmpeg 8.0.1) — «при воспроизведении таймлайн прокручивается, но тишина». В логе для канала `retro-pgm3` (wav/PCM, SMB pipe-режим):
```
[wav @ ...] invalid start code [254][250][231][250] in RIFF header
[in#0 ...] Error opening input: Invalid data found when processing input
pipe segment produced no output: 2026-07-14/14-18-55.wav  rc=183  byte_off=2446508  fine_ss=0.0
```

**Причина:** это отдельный, ранее существовавший баг в `_pipe_smb_segment()`/`_byte_offset_for_seek()`, не связанный с недавним фиксом тишины между фрагментами. Для WAV `_byte_offset_for_seek()` вычисляет байтовое смещение как `44 (заголовок) + сэмплы` и сразу сикает файловый хендл на эту позицию — то есть **пропускает RIFF-заголовок** и отдаёт в stdin ffmpeg сырые PCM-байты. Но `_pipe_smb_segment()` при этом всегда объявлял входной формат как `-f wav`, а WAV-демуксер ffmpeg требует настоящий RIFF/WAVE-заголовок в начале потока — на безголовом PCM он немедленно падает с ошибкой парсинга, и сегмент отдаёт вообще **ноль байт**. Значит, при любой перемотке ВНУТРЬ уже частично прослушанного WAV-фрагмента (`ss > 0`, обычная ситуация — ровно то, что происходит на каждой перемотке или старте воспроизведения не с начала файла) первый сегмент стрима полностью пропадал: таймлайн (который двигается независимо от контента, по элапсед-времени) прокручивался как обычно, а звука не было до следующего файла с `ss=0`. Смещение при `ss=0` (чтение с самого начала файла, где заголовок реально есть) эту проблему не имело — поэтому не все перемотки были одинаково «немыми».

**Исправление** (`app/audio.py`, `_pipe_smb_segment()`): когда для WAV byte_offset получился больше 0 (то есть реально сикнули мимо заголовка), входной формат переключается с `-f wav` на `-f s16le -ar <sample_rate> -ac 2` — это raw PCM демуксер, которому заголовок не нужен вообще, только явно заданные параметры (частота дискретизации канала и 2 канала, что соответствует формуле смещения). При `byte_offset == 0` (первый сегмент читается с реального начала файла, либо сработал overshoot-фолбэк на 0) `-f wav` остаётся как есть, так как там заголовок действительно на месте.

Проверено на реальных байтах фрагмента с проблемного канала (файл взят напрямую с шары): симуляция сикнутого на 50-й секунде запроса через старую логику (`-f wav` на безголовом PCM) воспроизводит ровно ту же ошибку `invalid start code ... in RIFF header`, что и в присланном логе; через новую логику (`-f s16le -ar 48000 -ac 2`) ffmpeg корректно декодирует и отдаёт `100.01` с — ровно `150.01 − 50` (реальная длительность файла минус величина сика), без единой ошибки.

### Продолжение: после фикса выше — «звук нормальный с начала файла, но искажения/шум при seek»

**Симптом:** пользователь прислал обновлённый лог того же продакшн-запуска. Формат-фикс выше сработал — ffmpeg действительно распознаёт вход как `s16le` (`Input #0, s16le, from 'pipe:0'`), сегмент больше не падает с ошибкой. Но при перемотке ВНУТРЬ фрагмента (`ss > 0`) звук — шум/искажения; воспроизведение с начала файла (`ss=0`) — чистое.

**Причина:** `_byte_offset_for_seek()` считает смещение как `44 + int(sample_rate * 2 * 2 * ss)`, где `ss` — разность Unix-таймстампов с плавающей точкой произвольной точности (например `7.462014583...`), а не аккуратная доля секунды. `sample_rate * 4 * ss` в общем случае **не является кратным 4** (4 байта = один стерео-фрейм из двух 16-битных сэмплов), и `int()`-усечение почти никогда не попадает на границу фрейма. Из реального лога: `byte_off=1432747` → `1432747 % 4 == 3`. Пока формат был `-f wav` (до предыдущего фикса), некорректный вход просто отклонялся ffmpeg целиком — сдвиг на 1-3 байта никогда не успевал проявиться как искажение. После перехода на `-f s16le` ffmpeg охотно декодирует ЛЮБОЕ выравнивание: смещённый на несколько байт поток трактуется как правильные 16-битные сэмплы, но фактически через один канал утекают старшие/младшие байты соседнего сэмпла — результат слышен как шум/искажение на всём протяжении сегмента (не только в начале), поскольку raw PCM не содержит самосинхронизирующихся маркеров кадров.

**Исправление** (`app/audio.py`, `_byte_offset_for_seek()`): вычисленное количество PCM-байт округляется вниз до кратного 4 (`raw -= raw % 4`) перед прибавлением 44-байтного заголовка; поскольку 44 само кратно 4, итоговое смещение остаётся кратным 4. Максимальная погрешность округления — 3 байта, то есть меньше одного сэмпла (~0.02 мс при 48 кГц) — неощутимо и не связано с искажением, о котором сообщил пользователь.

Проверено: фаззинг `_byte_offset_for_seek()` с 20 случайными `ss` (включая суб-микросекундные доли, как в реальных таймстампах) — все 20 итоговых смещений теперь кратны 4 (`(offset − 44) % 4 == 0`), до фикса воспроизводился случай из лога (`% 4 == 3`).

## 17. Индикаторы статуса ≤20 точек в строке; плейлоги сканируются по одному; «Выход» объединён с именем пользователя

*Сессия 2026-07-15, v1.2.090.*

### Индикаторы статуса: не более 20 точек в строке

`.status-dots` — общий CSS-класс для точек и каналов (`#status-dots`), и плейлогов (`#plbar-dots`) в статусной панели. Раньше ряд переносился по остатку доступной ширины flex-контейнера (`max-width: 60%`), поэтому число точек в строке плавало в зависимости от ширины окна и длины соседнего текста. `flex-wrap: wrap` остаётся — при нехватке места ряд переносится раньше (это не противоречит «не более 20»), но 21-я точка в любом случае уходит на новую строку.

### Сканирование плейлогов — по одному источнику за раз, как аудио

`_preload_playlogs()` (`app/main.py`) технически уже проверял плейлоги последовательно, но UI показывал это иначе: одно широковещательное `playlog_checking` без данных сразу помечало **все** точки как «проверяется», а один финальный `playlog_status` в конце разом проставлял всем результат — выглядело как параллельная проверка. Переписано по аналогии с `file_index.initial_scan` (который уже слал `index_scanning`/`index_progress` по одному каналу за раз):

- на каждый плейлог по очереди — `playlog_checking` (кто проверяется сейчас) → `playlog_progress` (результат по этому плейлогу);
- добавлен `asyncio.Lock` (`_playlog_scanning_lock`), сериализующий повторные запуски (старт / `POST /api/reload` / `POST /api/rescan_playlogs`) — раньше они могли реально пересекаться (гонка за одни и те же SMB-источники), теперь встают в очередь.

Frontend (`static/app.js`): `_handlePlaylogChecking(msg)` помечает «проверяется» только текущий плейлог (не все сразу); новая `_handlePlaylogProgress(msg)` инкрементально дополняет точки по мере прихода результатов. Добавлен i18n-ключ `playlog.checking_progress` (`ru`/`en`/`fr`).

Проверено изолированным скриптом, монтирующим реальный `app.main` с заглушками вместо `preload`/`check_sources` (без обращения к реальным SMB-источникам): подтверждена строгая последовательность сообщений по каждому плейлогу и то, что два параллельных вызова `_preload_playlogs()` не пересекаются благодаря блокировке.

### UI: пункт «Выход» объединён с именем пользователя в одну строку

`#menu-userinfo` в гамбургер-меню было две отдельные строки: имя пользователя и, ниже, полноширинная кнопка «🚪 Выйти». Объединено в одну flex-строку — иконка и имя слева, компактная кнопка «Выйти» справа (`static/index.html`, `static/style.css`: `.menu-userinfo-left`, `.menu-userinfo-logout`).

## 18. Точки статуса — 50 в ряд с группировкой; порядок точек плейлогов не «прыгал» при сканировании

*Сессия 2026-07-16, v1.2.091.*

### Диагностика: «не отображаются плейлоги Rock FM, Relax FM»

По жалобе на непоказанные плейлоги станции «101 Москва HLS» рабочие конфиги (`config/stations`, `config/playlogs`, `cache_playlogs`) были синхронизированы с продакшн-машиной для локальной проверки. Прямым вызовом `playlist.get_entries()` с реальными конфигами подтверждено, что `rock_fm` и `relax_fm` отдают корректные записи (362 и 282 трека за проверенную дату) — сами эти два плейлога были уже исправны. Обнаружена реальная причина остаточной проблемы: 4 канала станции (`business_fm`, `radio_orfey`, `radio_shanson`, `serebryanyy_dozhd`) в `config/stations/*.yaml` ссылаются на id плейлогов, для которых нет файла в `config/playlogs/` — такие id молча пропускаются в `load_playlists()` и никогда не попадают в `playlists_map`, поэтому не сканируются и не показывают точку статуса. Уточнено у пользователя: для этих 4 каналов (и отдельно для `novoe_radio`/`kalina`) метаданных не существует в принципе — плейлог-конфиги для них создавать не нужно, это ожидаемое отсутствие данных, а не баг.

### Индикаторы статуса: 20 → 50 точек в строке, группировка по 10

`.status-dots` (`static/style.css`) — лимит точек в строке увеличен с ≤20 (см. [§17](#17-индикаторы-статуса-≤20-точек-в-строке-плейлоги-сканируются-по-одному-выход-объединён-с-именем-пользователя)) до ≤50; для удобного визуального подсчёта каждые 10 точек разделены увеличенным промежутком:

```css
.status-dot { margin-right: 4px; }
.status-dot:last-child { margin-right: 0; }
.status-dot:nth-child(10n):not(:last-child) { margin-right: 12px; }
```

Раньше межточечный интервал задавался общим `gap: 4px` на контейнере — он не позволяет варьировать промежуток между отдельными точками, поэтому заменён на `margin-right` для каждой точки. `max-width` контейнера пересчитан под новую раскладку: `calc(50 * 8px + 49 * 4px + 4 * 8px)`.

### fix: точки плейлогов меняли порядок и «синяя» всегда была слева

При сканировании плейлогов `_handlePlaylogProgress()` (`static/app.js`) по завершении каждого источника удалял его точки из `_plSources` и добавлял их **в конец массива** (`.push(...)`), вместо обновления на месте. По мере прохода по списку уже проверенные плейлоги накапливались в хвосте массива, а ещё не проверенные оставались в исходном порядке в его начале — из-за этого весь ряд точек непрерывно «перетасовывался» во время сканирования, и индикатор «проверяется сейчас» (синий, мигающий), выставляемый на первый ещё не обработанный элемент, всегда оказывался самым левым. Исправлено: точки обновляются со splice на исходную позицию в массиве, порядок точек больше не меняется в процессе сканирования.

## 19. fix: точка индикатора статуса переносилась на новую строку некрасиво

*Сессия 2026-07-16, v1.2.092.*

### Раскладка точек: явное разбиение на ряды вместо `flex-wrap`

Пользователь заметил на скриншоте, что в ряду индикаторов одна точка «отрывалась» от остальных и переносилась на следующую строку сама по себе, ломая выравнивание. Причина: `max-width` на `.status-dots` (`static/style.css`) лишь ограничивает *максимальную* ширину контейнера — при нехватке реальной доступной ширины (соседние кнопки, скроллбар, масштаб страницы) браузерный `flex-wrap` мог перенести ряд на 1-2 точки раньше расчётных 50, и лишняя точка повисала в одиночестве.

Сначала на утверждение показан текстовый мокап раскладки (через `AskUserQuestion` с превью) — уточнено, что при переполнении неполный ряд должен быть **снизу**, а не сверху.

Раскладка теперь строится явно в JS, а не отдаётся на волю переноса браузера:

- Новый хелпер `_layoutDotRows(dotEls)` (`static/app.js`) делит точки на ряды по формуле `fullRows = floor(n / 50)`, `remainder = n % 50` — сначала полные ряды по 50, неполный (если есть) добавляется последним;
- внутри ряда группы по 10 считаются **с конца**: `firstGroup = size % 10 || 10` — так неполная группа всегда самая левая, а крайняя правая группа — всегда полная десятка;
- `.status-dots` (`static/style.css`) стал `flex-direction: column; align-items: flex-end` (колонка рядов) вместо `flex-wrap` + `max-width`; каждый ряд — новый `.status-dots-row` (`display: flex; justify-content: flex-end`);
- граница группы (увеличенный отступ) больше не считается CSS-селектором `nth-child(10n)` (он предполагает отсчёт группами по 10 от начала — ломается на неполном ряду), а размечается JS-классом `.group-end` на последней точке каждой группы.

Применено в обоих местах, где строится ряд точек: `_rebuildDots()` (статус каналов) и `_rebuildPlDots()` (статус плейлогов) — создание самих точек (классы, подсказки, обработчики кликов) не менялось.

## 20. Комплексный code-review: 10 high-severity фиксов

*Сессия 2026-07-18, v1.2.092 (pre-commit хук автобампа версии не установлен в этом окружении — `.git/hooks/pre-commit` отсутствует, только `.sample`-файлы, — поэтому VERSION в этом коммите не изменилась).*

По запросу «Сделай code-review, покажи отчёт» проведён обзор всей кодовой базы (не только диффа): четыре параллельных агента (security-reviewer, fastapi-reviewer, python-reviewer, typescript-reviewer) прошли по `app/` и `static/` независимо по разным углам — auth/access-control, async-корректность, вспомогательные модули, фронтенд-JS. Найдено 26 подтверждённых/правдоподобных проблем; два независимых прохода сошлись на одном и том же критическом баге (`/ws` без аутентификации), что подтвердило находку.

По запросу пользователя создана ветка `code/review`, в неё внесены исправления 10 high-severity находок:

1. **`/ws` полностью без аутентификации** (`app/main.py`) — `@app.middleware("http")` в Starlette не перехватывает websocket-апгрейды, поэтому ветка `path.startswith("/ws")` в `_auth_middleware` была мёртвым кодом. Добавлена явная проверка сессии внутри `websocket_endpoint` перед `ws.accept()`.
2. **Stored XSS в модалке проверки обновлений** (`static/app.js`, `_showUpdateModal`) — ветка «уже актуально» вставляла commit-message через `innerHTML` без экранирования, в отличие от ветки «есть обновление», которая уже использовала `_escHtml()`. Добавлено экранирование.
3. **Отсутствие rate-limiting на `/api/auth/login`** (`app/main.py`) — добавлен in-memory троттлинг по IP (5 попыток / 60 секунд), не позволяющий ни перебор пароля, ни спрей по AD-доменам с риском лока учёток.
4. **Fail-open при пустом пароле** (`app/auth.py`, `_check_local_password`) — запись `users.yaml` без `password`/`password_pbkdf2` давала `compare_digest("", "")` → `True` для пустого пароля. Теперь явный отказ, если пароль не задан.
5. **Блокирующее SMB-чтение в async-роуте `/api/playlist/{channel_id}`** (`app/main.py`) — `get_entries()` синхронно ходит в SMB и замораживала весь event loop на время SMB round-trip. Обёрнуто в `asyncio.to_thread`.
6. **Гонка на `SortedList` индекса файлов** (`app/file_index.py`, `ChannelIndex`) — конкурентная мутация из poll-loop (executor-поток) и чтение из event-loop потока без какой-либо синхронизации. Добавлен `threading.Lock`, оборачивающий чтение (`get_intervals`, `files_for_range`) и мутацию (`refresh`, `clear`).
7. **Утечка фоновых задач при `/api/reload`** (`app/main.py`, `app/file_index.py`) — каждый reload запускал новый набор poll/preload/refresh-циклов, не отменяя предыдущий; после N reload'ов работали N параллельных циклов на одних и тех же каналах. `_background_startup()` и `FileIndexManager.start_polling()` теперь отменяют предыдущие задачи перед запуском новых.
8. **Блокирующие `subprocess`/`urllib` в admin-роутах** (`app/main.py`, `check_updates`, `update_server`) — `git pull` (таймаут 60с) и GitHub API запрос выполнялись синхронно в async-функции, замораживая event loop для всех клиентов. Обёрнуто в `asyncio.to_thread`.
9. **Перечисление логинов через сообщения об ошибке** (`app/auth.py`, `authenticate`) — «неверный пароль» и «пользователь не найден» различались текстом, позволяя угадывать существующие локальные логины. Сообщения унифицированы.
10. **`/api/rescan_playlogs/{channel_id}` и `/api/rescan/{channel_id}` без `_require_admin()`** (`app/main.py`) — эндпоинты были доступны любому авторизованному не-админу, хотя фронтенд (`static/index.html`, `data-admin-only`) и README уже документировали их как admin-only. Добавлен `_require_admin(request)`.

Оставшиеся 16 low/medium-severity находок (утечка SMB-хендла при неудачном спавне ffmpeg, недоудалённый частичный файл экспорта при ошибке ffmpeg, отсутствующие WS-события `export_progress`/`export_done`/`export_error`, гонка при быстром переключении каналов в `loadPlaylist()`, отсутствие ресинхронизации состояния после переподключения WebSocket и др.) оставлены пользователю списком без исправления — приоритет отдан критическим и high-severity проблемам.

Верификация: `python3 -m py_compile` на изменённых `.py`-файлах, `node --check` на `static/app.js` — оба чисто. Полный запуск сервера не проверялся: в окружении нет установленного venv/зависимостей (`fastapi` и т.д. не установлены системно), поднимать окружение для разовой проверки не стали — изменения прослежены вручную по каждому call site и импортёру перед правкой.
